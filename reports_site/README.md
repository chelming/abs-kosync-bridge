# BookBridge Bug-Report Analytics Site

Internal MkDocs (Material) site that renders bug reports generated by the
diagnostics parser, with maintainer responses and submitter comments on every
report. This is **separate from the public docs site** at the repo root and is
never published — local/self-hosted viewing only.

## Structure

```
reports_site/
├── mkdocs.yml           # Material theme, search, auto-nav (no nav key)
├── requirements.txt     # mkdocs-material
├── Dockerfile           # mkdocs serve on :8000 (auto-rebuilds on change)
├── compose.yaml         # host port 5761, bind-mounts ./docs read-only
├── respond-finding.ps1  # PowerShell helper to post a maintainer response
├── report_template.md   # canonical per-report template (not rendered)
└── docs/
    ├── index.md         # homepage
    ├── findings.md      # live copy of DIAGNOSTICS_FINDINGS.md (generated)
    └── reports/
        ├── index.md     # reports landing page
        └── *.md         # one page per finding (generated, fully overwritten each scan)
```

## Parser integration contract

1. **Output directory:** `reports_site/docs/reports/`
2. **Filename:** `YYYY-MM-DD_<finding-id>.md` (date = the finding's first-seen date).
3. **Full regeneration:** every scan fetches the authenticated admin detail for
   each active finding and **overwrites** its report page with current data.
   This ensures summary, context, integrations, evidence, maintainer response,
   and submitter comments are always up to date. A temporary detail-fetch
   failure preserves the previous page because the filename is added to the
   expected set before the fetch is attempted.
4. **Stale-page cleanup:** after processing, generated files whose basename
   matches `^\d{4}-\d{2}-\d{2}_f\d+\.md$` and is not in the expected set are
   deleted. The `index.md` and committed sample (`2026-07-16_sample-finding.md`)
   are never touched.
5. **Digest:** after each scan, copy `docs/automated-review/DIAGNOSTICS_FINDINGS.md`
   to `reports_site/docs/findings.md` (overwrite allowed — it carries no
   developer responses).
6. **No nav edits needed:** navigation auto-builds from the file tree.

**Implemented:** `Sync-ReportsSite` in `scripts/automated-review/run-diagnostics-scan.ps1`
runs on every scan and always regenerates active finding pages. Each page
includes current summary, context, integrations, evidence, maintainer response
(from `response_md` / `response_at` fields on the finding), and submitter
comments (non-hidden, chronological).

## Posting a response

Use the included PowerShell helper to post a maintainer response to a finding:

```powershell
powershell -ExecutionPolicy Bypass -File reports_site/respond-finding.ps1 <id> "<message>"
```

The script PATCHes the receiver API with `{response_md: ...}`. The next
diagnostics scan regenerates the report page with the response included.

## Local-only warning

Generated content (`docs/findings.md`, `docs/reports/*` except `index.md` and
the sample) is gitignored so instance logs, responses, and comments never reach
the public repo.

## Run it

Local (auto-rebuilds on every parser write):

```bash
pip install -r reports_site/requirements.txt
mkdocs serve -f reports_site/mkdocs.yml   # http://127.0.0.1:8000
```

Static build:

```bash
mkdocs build -f reports_site/mkdocs.yml   # outputs reports_site/site/
```

Docker (from `reports_site/`):

```bash
docker compose up -d --build              # http://localhost:5761
```

The container runs `mkdocs serve` with `docs/` bind-mounted read-only, so new
parser output appears without a restart. For a hardened static alternative,
run `mkdocs build` and serve `site/` with any static file server, rebuilding
after each scan.
